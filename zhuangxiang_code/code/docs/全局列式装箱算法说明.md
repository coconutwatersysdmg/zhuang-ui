# 全局列式装箱 + 柱级组合优化（GCP）主算法说明

> 面向甲方工程人员与后续维护者：讲清楚新主装箱算法「怎么做、为什么这样、在哪里改、怎么开关」。

---

## 1. 它解决什么问题

旧的 beam search 逐盘贪心装箱，对「理论可整齐拼满」的规则订单**装填密度不足**：本能装到 192 指数（达标）的托盘只装到一半就另开新盘。例如 668 箱单订单（每盘理论 0.92 满载即达标），旧算法 20 盘仅 2 达标。

新主算法 **GCP（Global Column Packing）** 把装箱转化为**组合优化问题**求解，对规则数据达标率大幅提升（668 单订单：2 → 10 达标）。

---

## 2. 核心思想：两步降维 + 组合优化

装箱是 NP-hard。GCP 利用本问题的结构做两步降维，把它变成可精确求解的离散问题：

### 第一步：凑柱（3D → 2D）
同底面的箱子按高度叠成「柱」（高度和 ≤ 托盘高 720）。
- 柱从地面长起、同底面垂直堆叠 → **支撑率天然 = 1.0**（上箱完全压在下箱上）。
- 高度任意（**不要求 120 的倍数**）：规则箱凑满柱，不规则箱凑次满柱，都是合法柱。
- 于是每个箱的 3D 位置退化成「柱的 2D 底面位置 + 柱内 z 顺序」。

### 第二步：柱级组合优化（盘 = 柱的集合）
问题变成「把柱分配到若干托盘，每盘①柱底面能 2D 装入 1440×2240、②指数和 ≥192，**最大化达标盘数**」——这是经典的 **Set-Partitioning（集合划分）** 问题。
- **小规模分组**（柱类型少，如规则单订单）：用 OR-Tools **CP-SAT 精确 ILP** 求全局最优（668 单订单 0.2 秒解出 11 盘理论最优）。
- **大规模分组**（柱类型多）：用「同类满盘 + 贪心混合」快速构造（避免组合爆炸）。

### 落地（265 网格优先 + CP-SAT 精确摆柱兜底）
选中的每个达标盘**先用 265-单元列网格**定坐标（快、天然无缝）；网格因量化损失装不下时（如 93% 填充的混合底面订单差一两根、必须旋转才能塞下），改用 **CP-SAT 2D 精确摆柱**：允许 90° 旋转 + 混合列宽，坐标 ÷5 无损缩放，目标最大化装入指数；用网格的部分解作 **warm-start** 提示，稳定快速找到满装解（消除多线程在临界密度下的随机波动），落地后整体居中改善重心。逐箱复用 `SuctionPlanner` 生成吸盘姿态；残料柱拆回箱子交给 beam 装箱兜底。

---

## 3. 自适应：规则组 GCP，复杂组回退

真实订单常混有不规则箱。GCP 对「非满柱 + 大底面」的柱无能为力（这种柱满盘都到不了 192，柱化注定不达标）。因此 GCP **逐组自适应判定**：

- `GlobalColumnPacker.suits_group(boxes, target)`：凑柱后，若「有达标希望」的柱指数占比 ≥ 70%（规则数据特征），返回 True 走 GCP；否则返回 False，该组**自动回退旧 beam + 配方优先 + 救援链**，保证不退步。
- 实测 5000 箱（3 大组，含大量不规则箱）：规则成分高的组走 GCP，不规则组回退 baseline，总达标 = 旧算法持平（零回归），端到端 199 秒。

### 非模数箱型的朝向修正（去模数化）

`suits_group` 判定「每盘能装几根柱」原先用 265-单元列网格估算，对**非模数底面**
（长宽不是 350/265/530/700 的箱型，如 430×280、740×330）会严重低估
（430×280：量化估 16 根，真实可装 25 根）→ 误判「不适合列式」→ 把本可旋转装满的
订单回退到固定朝向 baseline、双重失效。现改用 `_orient_per`（真实 floor 几何根数，
取两种朝向中较优者）：模数箱型估值与网格**完全相等**（668/5000 零回归），非模数
箱型不再被低估——这类旋转敏感订单得以正确走 GCP + CP-SAT（CP-SAT 落地本就支持
90° 旋转），如纯 430×280 订单默认路径达标 0 → 9。

## 4. 约束如何保证

| 约束 | 保证方式 |
|---|---|
| 支撑率 0.8 | 柱内同底面堆叠 = 1.0；落地后整盘门禁复核 |
| 允许旋转 | 落地时每柱选最优朝向；旋转箱打 `layered_oriented` 标记（输出阶段保留旋转尺寸，避免间隙误判，对外输出时剥离该标记） |
| 吸盘可达 600×800 | 按 z 从低到高放置（放时上方无遮挡），`SuctionPlanner` 求姿态；门禁复核 |
| 间隙 6mm / 重心 1/3 | 柱紧贴排列（2mm < 6mm）、盘面均布；残料盘整体居中；门禁复核 |
| 小箱在下 / 同尺寸重箱在下 | 柱内按高度/重量排序（大、重在下） |

**所有盘最终都过 `validate_pallet_constraints` 整盘门禁**；残料盘走 beam 装箱，放置时即逐箱校验全部约束，保证必过门禁。

> **达标盘免 gap（业务决策）**：gap（间隙）约束的本意是防止装箱算法偷懒、留大空隙导致装不满。当整盘指数已达标（≥目标）时说明已尽力装满，剩余空隙是高密度装载的**几何必然**（如 94.9% 占用率必有 ~5% 碎片），门禁不再以 gap 判失败（`validate_pallet_constraints(target_mpm=...)` 控制）。未达标盘仍查 gap（防偷懒）。支撑率/重心/吸盘/越界/重叠对所有盘**恒查**，物理稳定性不受影响。这让"必须旋转柱才能单盘达标、但旋转产生侧向台阶空隙"的高密度订单（如 668 改造的 11 订单中 001/009）得以达标。

## 5. 怎么开关与配置

`config/packing_config.yaml` 的 `constraints` 段：

```yaml
constraints:
  main_packer: gcp    # 'gcp'=新主算法（默认）; 'beam'=旧算法（回退/审计）
```

- `main_packer: gcp`（默认）：规则组走 GCP，复杂组自动回退 beam。
- `main_packer: beam`：完全用旧 beam + 配方优先 + 救援链（与历史结果一致，零回归保险）。

命令行：`python code/run_packing.py --config code/config/packing_config.yaml`

**一个 yaml 同时控制 算法 + 约束 + 数据集 + 运行模式**，改配置即可，无需改代码：

```yaml
run_mode: normal          # normal=普通单次装箱; incremental=增量装箱
incremental:
  source_file: selected_6000_boxes_9_1_concentrated_3orders.xlsx  # 增量三表文件(含"新增箱"表)
excel_data:
  source_file: 668箱子数据集.xlsx     # 普通模式数据集(相对 data/，改这行即切数据集)
constraints:
  main_packer: gcp                    # 算法开关(见上)
  # ... 约束开关/数值 ...
```

- **数据集**：`excel_data.source_file`（普通模式）/ `incremental.source_file`（增量模式），相对 `data/`；文件不存在自动回退内置默认并告警。
- **运行模式**：
  - `run_mode: normal`（默认）——普通单次装箱，读 `excel_data.source_file`。
  - `run_mode: incremental`——增量装箱，读 `incremental.source_file`（需含「新增箱」表）：先装初始单 → 保留达标盘 → 把未达标盘的箱 + 新增箱重排 → 合并；输出带 `mode=incremental` 与 `incremental` 统计段。
- **关键**：增量模式与普通模式**共用同一套装箱核心**（都经 `build_workflow` → `main_packer` 分流），所以算法（gcp/beam）与全部约束**对两种模式同时生效**；增量只是在外层编排两次装箱 + 合并。
- 增量的 5 文件批量回归测试仍由独立入口 `python code/run_incremental_packing.py` 运行（不受 `run_mode` 影响）。

## 6. 代码位置（在哪里改）

| 文件 | 职责 |
|---|---|
| `src/packing/global_column_packer.py` | GCP 主算法：凑柱、ILP、贪心、网格/CP-SAT 精确摆柱落地（`_cpsat_pack_2d`，warm-start）、残料兜底、自适应判定（`suits_group` 用 `_orient_per` 真实几何根数去模数化，非模数订单不误判回退） |
| `src/packing/layered_packer.py` | 复用的凑柱 `_ffd_columns` 与组装 `_assemble`（吸盘字段、几何口径；`gap` 参数支持 CP-SAT 紧贴落地） |
| `src/main/workflow.py` | 逐组分流：`suits_group` 判定 → GCP 或回退 beam；GCP 盘数远超理论时**爆盘回退** baseline（防退步） |
| `src/geometry/constraint_validator.py` | 整盘门禁；`target_mpm` 参数实现"达标盘免 gap" |
| `src/config/constraint_config.py` | `main_packer` 配置字段 |
| `run_packing.py` | 入口：`load_constraint_config`/`load_data_filepath`/`load_run_config` 读 yaml；`_run` 按 `run_mode` 分流普通/增量；`_run_incremental` 增量两阶段编排 |
| `src/incremental/service.py` | 增量编排：两次 `workflow_factory()` 装箱 + 合并 + 守恒门禁 |
| `tests/test_global_column_packer.py` | GCP 单测（契约/守恒/门禁/自适应判定） |

### 关键可调参数（`global_column_packer.py` 顶部常量）
- `_UNIT = 265.0`：列网格 y 向基本单元（匹配本项目底面公共边）。
- `_OVERFLOW = 12.0`：达标盘允许的指数溢出上限（控制候选规模）。
- `_ILP_MAX_TYPES = 14` / `_MAX_ENUM`：柱类型数 / 枚举空间超阈值则改走贪心（防组合爆炸）。
- `_ILP_TIME = 15.0`：单组 ILP 时间上限（秒）。
- `_CPSAT_TIME = 15.0`：单盘 CP-SAT 精确摆柱时间上限（秒，warm-start 后通常数秒内出解）。
- `suits_group` 的 `0.70`：判定规则组的达标柱占比阈值。
- `workflow._run_group_gcp` 的 `theo + 1`：爆盘回退阈值（GCP 盘数超过"理论盘数+1"即回退 baseline）。

## 7. 怎么扩展

- **新箱型底面**：`_fp_orient` 增加底面→朝向映射；`_col_units` 已按 ceil 单元安全兜底任意 ylen。
- **改达标/填充策略**：调 `_OVERFLOW`（溢出容忍）、`_greedy_mixed_boards`（贪心填充策略）。
- **换求解器/目标**：`_solve_ilp` 内是标准 Set-Partitioning，可改目标（如最小化盘数 / 加权）或换求解器。
- **完全换算法**：实现一个有 `pack_group(pallet_type, sales_order_no, boxes, target) -> (plan, runtime, diag)` 的类，在 workflow 按 `main_packer` 接入即可；旧路径用 `main_packer: beam` 永久保留。

## 8. 验证基线
- 单订单 668（`668箱子数据集.xlsx`）：**11 盘 11 达标**（CP-SAT 落地后达理论上限，原 10）、守恒 668、约 8 秒。
- 11 订单 668（`668箱子数据集2.xlsx`，每单 93% 填充、理论 1 盘达标）：**6 达标**（001/004/005/008/009/011；002/003/006/007/010 因 97.8% 超高密度 CP-SAT 装不满、爆盘回退 baseline）、守恒 668、不崩、约 100 秒。原 GCP 仅 4 达标且崩溃/盘数爆炸。
- 5000 箱（不规则历史数据）：自适应 141 达标（= golden，零回归）、守恒 5000、约 210 秒。
- 回退 `main_packer: beam`：与历史 golden 一致。
- 增量模式（`run_mode: incremental`，6000 箱 = 初始 5000 + 新增 1000）：228 盘 173 达标、门禁 0 违例、守恒 6000、约 400 秒（改 CP-SAT 落地后建议复跑确认）。

> 注：上述"超大箱主导"的不规则数据（5000/增量）会自适应回退 beam，达标即 beam 水平；GCP 的增益体现在规则、无超大箱的数据（如 668：2→10）。GCP 不适用的边界与原因见 `MEMORY`/项目记录（530×700×480 这类占盘 1/4 的大箱，列式几何无解，需整盘分层而非单柱叠顶——已验证单柱叠顶无收益）。
