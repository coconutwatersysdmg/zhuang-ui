# 箱子旋转泛化性评估与修复（2026-06-21~22）

## 背景与动机

668/5000 数据集只是测试用。真实项目订单箱型多样，不能只按这两个数据集设计算法。
本次目标：评估当前算法（baseline = beam + 配方优先；增强 = GCP 全局列式装箱）在
**任意订单**下能否做到「能达标的托盘都达标、不达标的尽量装满」，重点审计 baseline
**固定朝向**（不允许箱子 90° 旋转）是否在某些箱型配比下成为缺陷。

## 真实箱型分析（决定性证据）

从 BMS（包装物料主数据）筛出 240 种可用箱型（长、宽 ≤ 800，mpm > 0）：

- **97 种（40%）旋转敏感**：旋转后单层格数更大（如 740×330 固定 6 格/层 → 旋转 12 格/层）。
- **208 种（87%）非模数**：长宽不是 350/265/530/700 这种整除托盘的模数。

668/5000 恰好几乎全是模数箱型（旋转 +0），这是早先「旋转无收益」结论的过拟合来源。
**真实世界必然出现旋转敏感、非模数箱型**——固定朝向是真实缺陷。

## 评估方法：已知最优构造法

为避免「用贪心实验外推上界」的选择偏差，采用**构造已知最优**：
每个诊断订单按 `总指数 = 192 × N`（0 残料）构造 → **理论上界 = 最优 = N 个达标盘**，
每盘恰好 192 指数。箱型 100% 取自真实 BMS。算法达到 N = 最优，< N = 暴露缺陷。

工具：`code/temp/eval_gen/make_orders.py`（生成器）+ `run_eval.py`（对比 baseline/GCP），
订单在 `code/temp/eval_gen/orders/`。7 类场景，每个理论最优 = 10 达标盘。

## 诊断：两层缺陷

| 订单（每盘 192，最优 10） | 改前 baseline | 改前 GCP 默认 |
|------|:---:|:---:|
| S1 模数对照 350×265 | 10 | 10 |
| S2 模数旋转敏感 530×350 | **0** | 10 |
| S3a 非模数旋转敏感 740×330 | **0** | 10 |
| S3b 非模数+量化 430×280 | **0** | **0（回退）** |
| S4 混合 530×350+740×330 | 6 | 10 |
| S5a 极端纵横比 800×350 | **0** | 10 |
| S5b 伴层依赖 700×530 | 10 | 10 |
| 668 真实 | 5 | 11 |

- **L1 — baseline 固定朝向（系统性）**：旋转敏感订单全面失分（0~6/10），并多占 40% 托盘。
- **L2 — GCP 模数化硬编码（局部但致命）**：`suits_group` 用 `_grid_pack`（350 列 / 265 单元
  量化）估算每盘根数，对非模数箱型严重低估（430×280：量化 16 vs 真实 25）→ 误判「不适合
  列式」→ 回退固定朝向 baseline → 双重失效（S3b 默认路径 0/10）。

## 修复

### L2 — GCP 去模数化（`src/packing/global_column_packer.py`）

新增 `_orient_per(xl, yl, pallet_dims, tol)`：真实 floor 几何每盘根数，取两朝向较优。
`suits_group` 改用它替代 `_grid_pack` 量化估算。

- 模数箱型两者数学相等（350×265→32、530×350→16、700×530→8）→ **668/5000 零回归**。
- 非模数箱型不再被低估 → 不再误判回退，走 GCP + CP-SAT（已支持旋转）→ S3b **0→9**。

### L1 — baseline 门槛规整（`src/main/recipe_first.py`）

`pack_group_recipe_first` 入口对每个箱子调 `_oriented_for_pallet(box, pallet_dims, target_mpm)`，
选最优朝向（生成副本、保留 id，**不修改原 box → 不污染 GCP 路径**；GCP 走 `_run_group_gcp`
不经过本函数）。规整后的箱子进入整层规划 / 落地 / 救援全程。

**几何门槛（关键设计）**：仅当旋转能让该箱型「单类整层」从不达标跨到达标
（`cap_fix < target ≤ cap_rot`，cap = 每盘根数 × 层数 × mpm）时才换向。

- 精准救回旋转敏感订单（S2/S3a/S3b/S4/S5a）。
- 对「旋转也救不了的难组」（5000 主体：箱型要么固定已达标、要么旋转也 < target）**保持
  原朝向、不打乱已紧凑的布局** → 5000 盘数零副作用。

开关：`constraint_config.allow_box_rotation_90`（默认 True），`workflow.py` 调用时传入。

> **弯路与教训**：先尝试「规整版 vs 原版双跑取优」的棘轮策略 → 5000 盘数没降（199，因
> 「达标优先」在同组「达标 +1 但盘数 +18」时仍取规整）且极慢（781s）。**几何门槛**才是
> 正确解：不双跑、快、精准锁定「旋转能救达标」的箱型。

## 验证结果

| 订单 | 改后 baseline（L1 门槛） | 改后 GCP 默认（L1+L2） |
|------|:---:|:---:|
| S1~S2,S3a,S4,S5a,S5b | 全 10/10 | 全 10/10 |
| S3b 非模数+量化 | **8/10**（0→8，柱式/整层凑整损失） | **9/10**（0→9） |
| 668 真实 | **10/11**（5→10，盘数 14→12） | **11/11**（零回归） |
| 5000 真实 | — | **141 / 182 盘**（零回归，盘数无副作用） |

- 测试套件：**82 passed**（唯一 failed 是备份目录 `备份/6.7新增箱/...` 的 pre-existing 旧测试，
  断言 recipe_rebuilder 旧键名，与本次改动无关）。
- 守恒、门禁（支撑率/重心/吸盘/越界/重叠）全程恒查，物理稳定不受影响。

## 已知限制：CP-SAT 多线程可复现性

`_cpsat_pack_2d` / `_solve_ilp` 用 `num_search_workers = 8`（多线程）+ `random_seed = 42`。
多线程 + 时限截断在 OR-Tools 中**仍非确定**（worker 调度时序影响取到的解）→ 668 gcp 偶尔
在 10/11 之间波动（总盘数也有 ±1 波动）。

实测单线程（`num_search_workers = 1`）确定可复现，但**慢约 5 倍且解质量更差**
（668 8/11、S3b 0/10、5000 不可行），已否决。接受多线程 ±1 盘波动为已知限制。

## 结论

- **当前算法能解决「几何受限」的旋转敏感订单**：GCP（去模数化后）接管主路径，baseline
  （门槛规整后）作为兜底，旋转敏感订单 baseline 全救回、GCP 默认路径仅 S3b 9/10（柱式
  凑整损失）。
- **对已优化的模数数据零副作用**：668/5000 达标与盘数均零回归。
- **真实世界泛化能力已具备**：固定朝向缺陷已修复，箱子允许 90° 旋转（可配置开关）。
- 残余 1-2 盘差距（S3b）是 GCP 柱式凑整的固有损失（192 不被柱指数整除），非朝向问题。
