# 装箱算法 ↔ 可视化系统 对接与交付文档

> 适用场景：可视化系统已能读取本算法输出的 JSON 结果文件、模拟装箱过程，现需让可视化系统**触发本算法运行并产出该 JSON**。
> 涉及"可选/建议"的部分均已显式标注。

---

## 0. 对接方式

可视化系统以**子进程**方式运行一条命令，算法把结果写成 JSON 文件，可视化系统读取该文件即可：

```bash
python code/run_packing.py --config code/config/packing_config.yaml --out <结果输出路径>.json
```

- 输入：由 `--config` 指定的 YAML 决定（见 §4）。**注意 `run_mode`**：
  - `run_mode: normal` → 读 `excel_data.source_file`（普通单次装箱，**标准对接用这个**）。
  - `run_mode: incremental` → 读 `incremental.source_file`（增量装箱，三表 Excel）。
- 输出：`--out` 指定路径的 JSON（结构见 §5），即可视化系统当前已支持的结果文件格式。
- 这是**零新增代码**的对接路径，命令、输入、输出均为现有实现。

> ⚠️ **出厂配置注意**：当前 `code/config/packing_config.yaml` 的 `run_mode` 是 `incremental`、数据集指向较大的 5000/8000 箱文件，单次运行需数分钟。做标准"数据集→JSON"对接前，**先把 `run_mode` 改成 `normal`，并把 `excel_data.source_file` 改成目标数据集**（见 §3.2）。已实测：`normal` 模式跑 `668箱子数据集.xlsx` 约 10 秒产出合法 JSON。

如果可视化系统是 Python 且希望进程内直接调用（不落地中间文件），见 §6 的 `run_with_boxes` 方式。

## 2. 目录结构与交付清单

算法靠 `src/config/constants.py` 中 `__file__` 推导路径，**与运行时所在目录无关**，但目录层级必须保持：

```
zhuangxiang_code/              ← 项目根（PROJECT_ROOT）
├── code/
│   ├── run_packing.py         ← 算法入口（命令行）
│   ├── requirements.txt       ← 依赖清单
│   ├── config/
│   │   └── packing_config.yaml← 运行配置（算法/约束/数据集/模式）
│   ├── src/                   ← 算法源码（全部）
│   └── docs/                  ← 本文档所在
├── data/                      ← 输入 Excel 存放目录（DATA_DIR）
│   └── 668箱子数据集.xlsx      ← 示例数据集（可换成可视化侧自己的数据）
└── output/                    ← 默认输出目录（OUTPUT_DIR，自动创建）
```

关键路径（代码硬约束，见 `src/config/constants.py`）：

- `PROJECT_ROOT` = `code/` 的上一级 = `zhuangxiang_code/`
- `DATA_DIR` = `zhuangxiang_code/data/` —— **输入 Excel 必须放这里**
- `OUTPUT_DIR` = `zhuangxiang_code/output/` —— 默认文件输出目录，运行时自动创建

**交付清单（打包整个 `zhuangxiang_code/`，或至少以下内容）：**

1. `code/` 整个目录（源码 + 入口 + 配置 + requirements）
2. `data/` 目录及其中的输入 Excel（至少留一个示例数据集）
3. 空的 `output/` 目录（可不带，运行时会自动建）
4. 本文档 `code/docs/可视化系统对接交付文档.md`

## 3. 运行说明

### 3.1 基本命令

```bash
# 在 zhuangxiang_code/ 目录下执行（CWD 不影响数据路径，但 --out/--config 的相对路径相对 CWD）
python code/run_packing.py --config code/config/packing_config.yaml --out output/result.json
```

- `--config`：配置文件路径。省略时默认用 `code/config/packing_config.yaml`。
- `--out`：结果 JSON 输出路径。**省略 `--out` 时不写该文件**，但仍会在 `output/` 下生成带时间戳的 `packing_plan_<时间>.json`（见 §3.3）。
- 其它可选参数：`--max-boxes N`（只取前 N 个箱子，快速冒烟）、`--safe`（配方/基线双跑审计，慢）、`--profile`（性能分析）。

### 3.2 配置文件 `packing_config.yaml` 关键项

```yaml
run_mode: normal                    # 标准对接=normal(读 excel_data)；incremental=增量(读 incremental.source_file)
incremental:
  source_file: selected_8000_boxes_9_1_concentrated_3orders.xlsx  # 仅 incremental 模式生效
pallets:
  MH423C:                           # 托盘类型 → 目标指数（达标线）
    length: 1440.0
    width: 2240.0
    height: 720.0
    target_index: 192
constraints:
  main_packer: gcp                  # gcp=主算法(默认,推荐)；beam=旧算法(回退/审计)
  # ... 其它约束开关/数值（间隙、支撑率、吸盘、旋转等），一般保持默认 ...
excel_data:
  source_file: 668箱子数据集.xlsx     # normal 模式输入数据集（相对 data/ 目录）
  result_sheet: 最终挑选结果           # 箱子任务表 sheet 名
  bms_sheet: 包装物料主数据(BMS)       # 物料主数据 sheet 名（提供箱子指数）
```

**标准"数据集→JSON"对接步骤**（关键）：
1. 把 `run_mode` 设为 `normal`；
2. 把目标数据 Excel 放进 `data/`，把 `excel_data.source_file` 改成该文件名；
3. 跑 §0 的命令。

> `run_mode` 决定读哪个数据文件：`normal` 读 `excel_data.source_file`，`incremental` 读 `incremental.source_file`（且 `--max-boxes` 在 incremental 模式下不生效）。文件不存在时算法会告警并回退到内置默认数据集。**出厂值是 `incremental`，标准对接务必先改回 `normal`。**

### 3.3 运行产物

| 产物 | 触发条件 | 路径 |
|---|---|---|
| `--out` 指定的 JSON | 传了 `--out` | 你指定的路径（紧凑 JSON，无缩进） |
| `packing_plan_<时间戳>.json` | 默认（除非进程内用 `NullReportPersister`） | `output/`（带 `indent=2` 缩进） |
| `packing_plan_summary_<时间戳>.xlsx` | 同上 | `output/`（托盘级统计表） |

两个 JSON 内容结构完全一致（见 §5），仅缩进不同。可视化系统读哪个都行；推荐用 `--out` 指定的路径，便于固定文件名对接。

### 3.4 运行耗时（参考，真实数据）

- 668 箱规则数据集：约 8~15 秒。
- 5000 箱数据集：约 150~280 秒（数据规模/机器而定）。

> 真实数据单次运行可能持续数分钟，可视化系统若需即时响应，建议**异步触发 + 轮询结果文件是否生成完毕**。

## 4. 输入格式

### 4.1 文件式（Excel，CLI 默认路径）

输入是一个 `.xlsx`，需包含**两个 sheet**（见 `src/data/excel_loader.py`）：

**Sheet 1：箱子任务表**（默认名 `最终挑选结果`，可在 yaml 的 `result_sheet` 配）。每行一个箱子，列名固定：

| 列名 | 含义 | 必填 |
|---|---|---|
| `箱子序号` | 箱子唯一编号（重复编号会自动加 `__row<行号>` 后缀去重） | 是 |
| `Box类型` | 箱型/包装规格代码，用于到 BMS 表查指数 | 是 |
| `箱子长` / `箱子宽` / `箱子高` | 箱子尺寸（mm） | 是 |
| `总重量` | 重量（缺失按 0） | 建议 |
| `销售订单号` | 订单号（空值按 `UNKNOWN_ORDER`） | 是 |
| `Case类型` | 托盘类型（如 `MH423C`），决定目标指数 | 是 |
| `托盘长` / `托盘宽` / `托盘高` | 该箱所属托盘尺寸（mm） | 是 |

**Sheet 2：物料主数据**（默认名 `包装物料主数据(BMS)`，可在 yaml 的 `bms_sheet` 配）。算法据此为每个箱型取"指数"：

| 列名 | 含义 |
|---|---|
| `包装规格代码` | 与 Sheet1 的 `Box类型` 对应（作索引） |
| `最小包装量的倍数` | 该箱型的指数值（即 `min_pack_multiple`） |

> 分组规则：算法按 `(Case类型, 销售订单号)` 把箱子分组，逐组装箱。同一分组应使用同一套托盘尺寸。

### 4.2 进程内式（箱子列表，见 §6）

若不用 Excel，可直接构造箱子字典列表传入 `run_with_boxes(boxes)`。每个箱子字段：

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| `id` | str/int | 是 | 唯一 ID（同一次请求内不重复） |
| `type` | str | 是 | 箱型编码 |
| `length`/`width`/`height` | number | 是 | 箱子尺寸（mm，>0） |
| `weight` | number | 建议 | 重量（缺失按 0） |
| `min_pack_multiple` | number | 是 | 指数值 |
| `pallet_type` | str | 是 | 托盘类型（匹配目标指数） |
| `sales_order_no` | str | 是 | 销售订单号 |
| `pallet_dims` | object | 是 | `{length, width, height}` 托盘尺寸（mm） |

## 5. 输出 JSON 格式（可视化读取的结果文件）


### 5.1 顶层

```json
{
  "packing_plan_id": null,
  "total_runtime_seconds": 0.42,
  "summary": { "overall": {...}, "by_pallet_type": {...} },
  "pallets": [ ... ]
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `packing_plan_id` | null | 算法不生成业务计划号，恒为 `null` |
| `total_runtime_seconds` | number | 总耗时（秒） |
| `summary.overall` | object | 全局统计：`total_pallets / success_pallets / failed_pallets / unknown_pallets / avg_mpm_gap / max_mpm_gap / rescued_from_failed / kpi / runtime_breakdown_seconds` 等 |
| `summary.by_pallet_type` | object | 按 `{pallet_type}__{sales_order_no}` 的分组统计 |
| `pallets` | array | **托盘明细列表（可视化主要消费的部分）** |

### 5.2 单个托盘（`pallets[i]`）

```json
{
  "pallet_id": "MH423C-DEMO-ORDER-1",
  "pallet_type": "MH423C",
  "sales_order_no": "DEMO-ORDER",
  "mpm_total": 192.0,
  "mpm_target": 192,
  "mpm_gap": 0.0,
  "mpm_status": "SUCCESS",
  "stability_checks": { "status": "SUCCESS" },
  "box_total_volume": 1068480000.0,
  "pallet_volume": 2322432000.0,
  "fill_rate": 0.460069,
  "packed_items": [ ... ]
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `pallet_id` | str | `{pallet_type}-{sales_order_no}-{序号}` |
| `mpm_total` / `mpm_target` / `mpm_gap` | number | 指数合计 / 目标 / 缺口(`target-total`) |
| `mpm_status` | str | `SUCCESS`(达标) / `FAILED`(未达标) / `UNKNOWN`(该托盘类型未配目标指数) |
| `stability_checks.status` | str | `SUCCESS`/`FAILED`（重心稳定） |
| `box_total_volume` / `pallet_volume` / `fill_rate` | number | 体积统计（mm³）与填充率 |
| `packed_items` | array | 箱子摆放明细 |

### 5.3 单个箱子（`packed_items[j]`）—— 真实样例

```json
{
  "id": "BOX000",
  "type": "PKG-A",
  "length": 352.0,
  "width": 532.0,
  "height": 240.0,
  "weight": 5.0,
  "min_pack_multiple": 8.0,
  "pallet_type": "MH423C",
  "sales_order_no": "DEMO-ORDER",
  "pallet_dims": { "length": 1440.0, "width": 2240.0, "height": 720.0 },
  "raw_length": 350.0, "raw_width": 530.0, "raw_height": 240.0,
  "original_length": 350.0, "original_width": 530.0, "original_height": 240.0,
  "position": { "x": 0.0, "y": 0.0, "z": 0.0 },
  "volume": 44520000.0,
  "supported_area": 187264.0,
  "support_ratio": 1.0,
  "suction_box_corner": "x_min_y_min",
  "suction_cup_corner": "x_min_y_min",
  "suction_orientation": "cup_600x_800y",
  "suction_cup_x_size": 600.0, "suction_cup_y_size": 800.0,
  "suction_rect_x_min": 0.0, "suction_rect_x_max": 600.0,
  "suction_rect_y_min": 0.0, "suction_rect_y_max": 800.0
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` / `type` | str | 箱子 ID（对应输入）/ 箱型 |
| `position.x/y/z` | number | **箱子最小角（左下后角）坐标**，mm。原点=托盘底面 `x`长 / `y`宽 / `z`高 角 |
| `original_length/width/height` | number | **箱子真实尺寸**（已按摆放朝向给出，旋转已考虑） |
| `raw_length/width/height` | number | 同 `original_*`（内部口径，值相同） |
| `length/width/height` | number | **占位尺寸 = 真实尺寸 + 2mm 间隙容差**（`xy_tolerance=2`，z 容差=0） |
| `min_pack_multiple` | number | 该箱指数 |
| `volume` | number | 真实体积（mm³） |
| `supported_area` / `support_ratio` | number | 直接支撑面积 / 支撑率 |
| `suction_*` | mixed | 机械臂吸盘姿态与吸盘矩形（600×800），供机器人侧使用 |

## 6. 可视化渲染指引（关键，避免画错）

坐标系（见 `_assemble` 与 `output_formatter`）：

- 原点 `(0,0,0)` = 托盘底面"左下后"角。
- `x` 沿托盘**长**方向（`pallet_dims.length`，如 1440）。
- `y` 沿托盘**宽**方向（`pallet_dims.width`，如 2240）。
- `z` 沿**高**方向（`pallet_dims.height`，如 720）。
- 单位统一 mm。

**画托盘**：在原点画一个 `pallet_dims.length × width × height` 的底座/边框。

**画每个箱子**：在 `(position.x, position.y, position.z)` 处画一个尺寸为
`original_length × original_width × original_height` 的长方体（`original_*` 沿 x/y/z）。

> ⚠️ **不要用 `length/width/height` 画箱子**——它们是"真实尺寸 + 2mm 间隙"的占位框（碰撞盒），直接画会让箱子之间看起来粘连/略大。`length=352` 而真实 `original_length=350`，2mm 是算法预留的间隙容差。可视化要么用 `original_*`（真实箱、箱间留 2mm 缝），要么用 `length/*`（占位盒、紧贴）。**渲染真实箱体请用 `original_*`。**

**旋转处理**：若箱子被算法旋转 90°，`original_length/width` 已经是**摆放后**的朝向（旋转已写入，无需可视化再转）。直接 `original_length` 当作 x 方向边长即可。

**装箱过程动画（自底向上、从角点扩散）**：对一个托盘的 `packed_items` 按 `(position.z, position.y, position.x)` 升序排序后逐个"落箱"，即可还原算法的码垛顺序（先底层、每层从 `(0,0)` 角向外）。这与 `_assemble` 内部的放置顺序一致。

**吸盘（可选）**：如需展示机械臂抓取，可在每个箱子顶面按 `suction_rect_x_min/x_max/y_min/y_max` 画 600×800 的吸盘矩形，`suction_orientation` 给出吸盘朝向。

## 8. 质量门禁与错误处理

算法返回前会跑输出质量门禁（`ResultFormatter.validate_output_quality`），下列情况**抛 `ValueError` 异常、不返回结果**：

- 出现空托盘；
- 输入箱子未全部出现在输出；
- 输出出现输入之外的箱子；
- 同一箱子重复出现在多个托盘；
- 箱子尺寸/体积为 0。

CLI 方式下异常会使进程非零退出并打印错误；进程内方式建议 `try/except ValueError` 捕获。可视化侧据此判断"本次装箱失败"。

## 9. 注意事项与边界

- **ortools 必须装**才能拿到算法的完整达标率；缺失仅作降级运行，不报错但效果差。
- 输入 Excel 的两个 sheet 名、列名要与 §4.1 一致（或在 yaml 中配 `result_sheet/bms_sheet`）。
- `data/` 路径由代码 `__file__` 推导，**目录层级不能改**（`code/` 必须在 `zhuangxiang_code/` 下，`data/`、`output/` 与 `code/` 同级）。
- `pallet_type` 必须在配置 `pallets:` 里有对应目标指数，否则该组 `mpm_status=UNKNOWN`。
- 大数据集运行耗时数分钟，建议异步触发 + 轮询结果文件。
- 已知限制：CP-SAT 多线程求解存在 ±1 个托盘的轻微波动，同一输入多次运行结果可能极小差异，属正常。
